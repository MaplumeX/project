## 二、需对照 TEC-8 ALU 功能表确认的问题

### 问题 #1 / #5：NOT 指令 `S` 编码（非 Bug，已回退为取反码）

**位置**：`pipelined_hardwired_controller.v`，指令 `4'b1101`（NOT）的 W1 分支。

**当前代码**（已显式设置 `S = 4'b0000`）：

```verilog
4'b1101: begin // NOT：执行按位取反并写回寄存器。
    if (W1) begin // 短周期：取反结果经 ABUS 写回寄存器，同时取下一条指令。
        SHORT = 1'b1;
        PCINC = 1'b1;
        LIR = 1'b1;
        DRW = 1'b1;
        ABUS = 1'b1;
        LDZ = 1'b1;
        M = 1'b1;
        S = 4'b0000; // 正逻辑逻辑模式(M=H): F = /A（按位取反）
    end
end
```

**历史与结论更正**：

本问题最初被判定为 Bug，依据是一份结论称"正逻辑逻辑列 `S=0000` 对应 `F=A`（传送），
取反需 `S=0100`"，并据此把 NOT 从未赋值（默认 `S=0000`）改为 `S=4'b0100`。
**该判定是功能表读反的错误**，实测 NOT 失效（R1 不变）。对照本项目 `74ls181.md`
所载 74181 功能表的**正逻辑-逻辑(M=H)**列：

| S3 S2 S1 S0 | 正逻辑-逻辑(M=H) | 含义 |
| --- | --- | --- |
| 0000 | $\overline{A}$ | **取反**（NOT 期望） |
| 0100 | $\overline{AB}$ | **NAND**（误改后的实际行为） |
| 1010 | B | 传 B（MOV/OUT/LD/ST 期望） |
| 1011 | $\overline{A+B}$ | —（注：AND 用算术列或另定） |
| 1111 | A | 传 A（JMP 期望） |

即 `S=0000` 在正逻辑逻辑列就是**取反**，NOT 最初依赖默认值 `0000` 其实是对的。
`S=0100` 是 NAND，且 NOT 是单操作数指令、B 未驱动，故误改后 NOT 自然失效。

**已应用的修复**（回退）：将 NOT 分支的 `S` 改回 `4'b0000`（取反），并显式写出，
不再依赖主控制块顶部的默认清零赋值。

> **前注记的修正**：原注记称"主控制块默认清零已删除、`S` 会综合为锁存器"不成立——
默认清零块（`S = 4'b0000;` 等行）实际仍在（见 `pipelined_hardwired_controller.v` 
`always @(*)` 块开头，与问题 #7 一致）。且 NOT 分支现已显式赋值 `S`，
不再依赖默认/锁存假设。

> **待实测确认**：请在 TEC-8 上用 `0FH/00H/FFH` 三组初值实测 NOT，确认
> R1 写回值分别为 `F0H/FFH/00H`，Z 标志在结果为 00H 时为 1。

### 问题 #6：DEC 指令 ALU 编码 `S / CIN`（基本无问题，建议二次确认）

**位置**：`pipelined_hardwired_controller.v`，指令 `4'b1111`（DEC）的 W1 分支。

**当前代码**（已优化为短周期，ALU 编码未变）：

```verilog
4'b1111: begin // DEC：执行自减并写回寄存器，同时更新 Z/C。
    if (W1) begin // 短周期：自减结果写回寄存器并更新标志位，同时取下一条指令。
        SHORT = 1'b1;
        PCINC = 1'b1;
        LIR = 1'b1;
        DRW = 1'b1;
        CIN = 1'b1;
        ABUS = 1'b1;
        LDZ = 1'b1;
        LDC = 1'b1;
        S = 4'b1111;
    end
end
```

**问题描述**：

DEC（自减，即 `A - 1`）的常用实现思路有两种：

1. **减法思路**：`A - 1 = A + (-1 的补码) = A + 1111 + 0`，使用减法功能码
   （参考 SUB：`S = 0110`），`CIN = 0`。
2. **加法思路**：`A - 1 = A + 1111 + 1 = A + (1111 + 1) 溢出回 0000`，使用
   加法功能码（参考 ADD：`S = 1001`），`CIN = 1`。

当前代码使用 `S = 1111` + `CIN = 1`。

**已查证依据**：

- TEC-8 摘选表中 `S=1111, M=0, CIN=1`（即 CN=1，无进位）对应 **`F = A - 1`**，
  与 DEC 语义吻合。
- 代码中 ADD 用 `CIN=1`、INC 用默认 `CIN=0`，反推 TEC-8 的 `CIN` 等价于 74181
  的 `/Cn`（低有效），与摘选表"CN=1 无进位"的标注一致。

**结论**：`S=1111, CIN=1` 在 TEC-8 中确实实现 `A - 1`，**基本确认不是 Bug**。
`LDC` 的借位语义也与 SUB 一致，合理。

**建议**：仍建议查完整表二次确认 `S=1111` 的算术运算定义。

## 三、默认清零赋值块的存废（已恢复，待与老师二次确认）

**位置**：`pipelined_hardwired_controller.v`，`always @(*)` 开头对全部输出做默认清零的
20 行赋值。

**背景**：commit `b5d62340` 曾按老师意见删除该块，理由是“各 case 分支已完整覆盖所有
输出，默认清零不必要”。删除后我又加回（见上方恢复记录）。

**分歧**：

- 老师认为：该默认清零块**不必要**，属于冗余写法。
- 实测现象：删除后组合逻辑块的未覆盖信号会被综合为**锁存器**（latch），保持上一拍值，
  导致行为随前置指令飘移。这与“组合逻辑不应推断锁存器”的常规约束一致。具体动因见
  问题 #1/#5：NOT 分支原先未赋 `S`，删除默认清零后 `S` 变成锁存器，实测依赖前置指令。

**结论**：已将默认清零块恢复。老师“不必要”的结论可能是基于“每个分支都完整赋值所有
输出”的假设，而当前代码并非每个分支都覆盖所有信号（例如 NOT 分支历史上就漏设 `S`）。

**待确认**：拿这个问题与老师对一次，明确二选一主张：

1. 保留默认清零块（当前做法，稳妥，无锁存器风险）。
2. 删除默认清零块，但要求**每个 case 分支显式覆盖所有输出**（需逐一审查补全）。

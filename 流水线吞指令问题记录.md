# TEC-8 流水线吞指令问题记录

## 问题现象

采用流水线指令周期的硬布线控制器,在 TEC-8 实验台上运行样例程序时,会"吞掉"指令——实际取到的是下下条指令,导致程序执行错乱。

## 根因

不是控制器逻辑 bug,而是 **TEC-8 的硬件固有问题**。

流水线设计将原非流水线中独立为一个节拍的取指操作(`LIR`、`PCINC`)合并到每条指令的执行节拍里,以重叠执行+取指。在原代码中,`LIR` 和 `PCINC` 由 **纯组合逻辑 `always @(*)`** 产生,二者在同一个节拍同时拉高。

但在 TEC-8 实验台上:

- `PCINC` 引脚到 PC 计数器、`LIR` 引脚到 IR 寄存器的 **布线/门延时不一致**;
- 出现 **反序触发**:正常顺序应先 `LIR`(锁存 PC 当前指向的指令)再 `PCINC`(PC+1),但实际是 `PCINC` 先生效 → PC 先 +1 → 存储器地址已变 → `LIR` 才锁存 → **取到下下条指令 → 吞指令**。

哪些指令会吞,取决于具体实验台的布线(因台而异)。北邮学长项目 [WadeChiang/CPU-Controller-TEC-8](https://github.com/WadeChiang/CPU-Controller-TEC-8) 记录了 SUB/LD 为主,自己测试中 ADD/JC/JZ 也有。

## 解决方案:A. 寄存器化 LIR/PCINC(已采用)

把 `LIR`/`PCINC` 从组合 `always @(*)` 中剥离,改为在 **`T3` 上升沿用寄存器同步锁存**,且二者的取指条件表达式 **完全相同**。这样两信号在同一个时钟沿翻转、严格同步建立,在 PC/IR 响应沿(下降沿)之前已稳定,消除反序触发。

参考实测可行实现:[nobugCPU `nobugCPU-pipe.vhd`](https://github.com/xqmmcqs/nobugCPU)。

### 关键代码(新增时序块)

```verilog
wire FETCH;
assign FETCH = (INS_FETCH && ST0) && (
           (W1 && (
               (IR7_IR4 == 4'b0000) ||                          // NOP
               (IR7_IR4 == 4'b0001) ||                          // ADD
               (IR7_IR4 == 4'b0010) ||                          // SUB
               (IR7_IR4 == 4'b0011) ||                          // AND
               (IR7_IR4 == 4'b0100) ||                          // INC
               (IR7_IR4 == 4'b0111 && !C) ||                    // JC 不跳转
               (IR7_IR4 == 4'b1000 && !Z) ||                    // JZ 不跳转
               (IR7_IR4 == 4'b1010) ||                          // OUT
               (IR7_IR4 == 4'b1011) ||                          // MOV
               (IR7_IR4 == 4'b1100) ||                          // CMP
               (IR7_IR4 == 4'b1101) ||                          // NOT
               (IR7_IR4 == 4'b1111)                            // DEC
           )) ||
           (W2 && (
               (IR7_IR4 == 4'b0101) ||                          // LD
               (IR7_IR4 == 4'b0110) ||                          // ST
               (IR7_IR4 == 4'b0111 && C) ||                     // JC 跳转成立
               (IR7_IR4 == 4'b1000 && Z) ||                     // JZ 跳转成立
               (IR7_IR4 == 4'b1001)                             // JMP
           ))
       );
always @(posedge T3 or negedge CLR) begin
    if (!CLR) begin
        PCINC <= 1'b0;
        LIR   <= 1'b0;
    end
    else begin
        PCINC <= FETCH;
        LIR   <= FETCH;   // 关键:与 PCINC 表达式完全相同
    end
end
```

### 取指时序规则

| 阶段 | 条件 | 取指时机 |
|---|---|---|
| 启动拍 | `INS_FETCH && !ST0 && W1` | 不取指(只装 PC,保留 SHORT 单拍设计) |
| 短指令 + JC/JZ 不跳 | NOP/ADD/SUB/AND/INC/OUT/MOV/CMP/NOT/DEC, JC&!C, JZ&!Z | `W1` 取指 |
| 长指令 + 条件跳转成立 + JMP | LD/ST, JC&C, JZ&Z, JMP | `W2` 取指 |
| STP / 手动模式 | — | 不取指 |

## 原理:为什么寄存器化能解决吞指令

核心在于 **组合输出的"电平"和寄存器输出的"沿"在时序契约上的根本差别**。

### 1. 组合输出为什么会产生反序

组合逻辑 `always @(*)` 产生的 `LIR`/`PCINC` 是**电平信号**:不是有确定时刻的跳变,而是"在节拍电位 W1/W2 期间维持为 1 的一段电平"。

`LIR` 和 `PCINC` 虽逻辑上应同时为 1,但它们是**两路独立的组合输出**,各自经不同门级路径和布线到达 PC 计数器、IR 寄存器输入端:

```
FETCH逻辑 ─┬─[门链 A, 延时 t_A]─→ PCINC ─→ PC 的 inc 输入
           └─[门链 B, 延时 t_B]─→ LIR   ─→ IR 的 load 输入
```

当 `t_A ≠ t_B` 时,这两个"电平"在时间上不是整齐对齐的边沿,而是**两段有先后、有偏斜的电平**。在 T3 响应沿(下降沿)附近的采样窗口里,可能:

- `PCINC` 的有效电平先覆盖 PC 触发器采样窗 → PC 先 +1 → 存储器地址已变;
- `LIR` 的有效电平稍后才到 → IR 锁存到的是 PC +1 后(或变化中)地址所指内容 → **取错指令**。

即学长所说的"按相反次序触发 LIR 和 PCINC"。组合逻辑里"哪个先达有效"由布线延时决定,**不确定、因台而异**——所以哪条指令吞也因台而异。

### 2. 寄存器输出为什么能消除反序

改成 `posedge T3` 寄存器后,`LIR`/`PCINC` 变成**时钟沿驱动的输出**:

```verilog
always @(posedge T3) begin
    PCINC <= FETCH;
    LIR   <= FETCH;   // 同一个 FETCH,同一个沿
end
```

关键变化有三层:

**(a) "何时变为有效"被固化到一个确定沿**
组合逻辑里信号"何时达到有效"由路径延时决定,两路不同步。寄存器化后,两路都从**同一 T3 上升沿**开始向有效跳变,起跳时刻由时钟统一驱动,不再各自漂移。

**(b) 有效电平覆盖 T3 整个高电平稳态区**
寄存器在 T3 上升沿锁存 `FETCH`,此后整个 T3 高电平期间 `PCINC` 和 `LIR` **稳定同为高,且严格相等**。即使寄存器到两目标引脚的 `ck→Q` 延时仍有微小差异,那也只是纳秒级固定偏差,不再有"谁先覆盖采样窗"的竞争窗口。

**(c) 与响应沿(下降沿)配合,天然保证"先取当前指令,后 PC+1"**

真正的取指动作发生在 **T3 下降沿**(PC/IR 的响应沿,与 `Q`、`ST0` 更新沿一致):

```
T3:  ___|‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾|___
         ↑寄存器锁存FETCH      ↑响应沿
         PCINC/LIR 变为 1      此时三者同为 1,稳定
              │                ↓此处:
              │ 存储器对当前 PC 地址组合输出指令(高电平期间已稳)
              │ ─────────────→ ↓LIR 锁存 → 取到"当前指令"
                                ↓PCINC → PC+1 指向"下一条"
```

因为:
- T3 高电平期间 PC 还没 +1,存储器输出当前指令,这段时间足够长且已稳定;
- 下降沿时刻,`LIR` 同沿把"当前指令"锁进 IR,`PCINC` 同沿把 PC +1 推向下一条;
- 两操作在**同一响应沿**完成,`LIR` 用的地址是"PC 尚未 +1 的那一刻",所以锁到的必然是当前指令,顺序上天然 **LIR 取当前 → PC+1 指向下条**,不可能反序。

### 3. 一句话对比

| | 组合输出 | 寄存器输出 |
|---|---|---|
| 信号形态 | 持续电平,起止时刻由路径延时决定 | 时钟沿驱动的跳变 |
| 两路对齐 | 各走各的路径,延时不等 → 有先后偏斜 | 同一 FETCH、同一时钟沿起跳 → 严格同步 |
| 响应沿附近 | 可能出现"PCINC 先有效、LIR 后有效"的竞争窗 | 全程稳定同为高,无竞争窗 |
| 取指顺序 | 可能反序 → 取到下下条 | 下降沿同沿锁存,天然"先取当前,后 +1" |

**寄存器化不改变逻辑功能,改变的是时序契约**:把"两路电平各自漂移、谁先谁后看布线"的不确定关系,变成"同一沿同步建立、在稳定区被同一响应沿采样"的确定关系——反序触发的温床被消除。



### 那顺序式为什么不出事?

   顺序式里 LIR/PCINC 同样是组合逻辑,理论上也能反序,但它没吞,是因为有两个裕量在兜底:

   1. 取指是独立专属节拍(W1),单一职责:那一拍数据通路基本闲着(没有 ABUS 写回、ALU 操作等),FETCH→{LIR,PCINC} 的组合路径极
      短,两者几乎由 W1 直接驱动,延时差可忽略到不触发反序。
   2. 节拍时间长、信号全程稳态:W1 维持一整个机器周期(含 T1/T2/T3),LIR/PCINC 在 T3 响应沿到来前已稳定很久,裕量充足。

   流水线合并后,这两层裕量同时被吃掉:
   - W1 同时驱动十几个信号(ABUS/DRW/ALU/S/CIN/LDZ/LDC/LIR/PCINC...),FETCH 进 LIR/PCINC 的路径变长变复杂,两者的生成路径差
     异被放大;
   - 执行操作和取指在同一拍重叠,数据通路总线、存储器端口切换引入额外不确定性。

   顺序式不是没有这个病,是裕量大把它盖住了。流水线合并把病逼出来了。

## 实测验证前提 ⚠️

本方案依赖一个时序假设(与 nobugCPU 一致):

> **PC / IR 在 `T3` 下降沿响应 `PCINC`/`LIR`**(与 `Q`、`ST0` 的更新沿一致)。

若某台实验台 PC/IR 改在 `T3` 上升沿响应,会形成"同沿锁存 + 同沿响应"的竞争。烧录后用样例程序跑一遍验证;若仍有吞指令,用单步/微单步观察 `PCINC`、`LIR` 在 T3 高/低电平期间的先后,若确认在上升沿响应,则把时序块改成 `negedge T3` 锁存。

## 备选方案 B(未采用)
